## 一、结论先行

| 问题 | 答案 |
|------|------|
| 账单是否会根据合同 `rent_due_day` 生成到期日？ | **是** — 生成账单时读取合同截止日，计算 `due_date` |
| 系统是否**每天自动**检测账单逾期？ | **否** — 没有定时任务，没有后台线程，没有 cron |
| 逾期判断逻辑在哪里执行？ | **被动查询** — 仅在首页 Dashboard API 被调用时实时 SQL 计算 |
| 是否存在自动催收？ | **否** — 催收完全依赖人工点击触发 |
| 是否有"逾期"状态持久化？ | **否** — 数据库无 `逾期` 状态，全靠 `due_date < now` 运行时计算 |

**核心结论：系统能正确判断逾期，但不会主动检测。** 每次打开首页时，Dashboard API 实时查询逾期账单数量。没有后台进程每天自动扫描、自动标记、自动催收。

## 二、账单生成：到期日如何计算

### 2.1 代码位置

**文件**：[`backend/services/bill_service.py`](backend/services/bill_service.py)

**函数**：`generate_bill()`（第 41 行起）

### 2.2 执行逻辑

```python
# 第 64-67 行
# 计算到期日：账单月份 + rent_due_day
ym = data.bill_month  # 例如 "2026-06"
year, month = int(ym[:4]), int(ym[5:7])
due_date = f"{year}-{month:02d}-{c.rent_due_day:02d}"
```

**步骤分解**：
1. 前端传入 `bill_month`（如 `"2026-06"`）
2. 从合同对象 `c` 读取 `rent_due_day`（如 `15`）
3. 拼接为 `"2026-06-15"` 存入 `Bill.due_date`

### 2.3 完整生成流程

```
用户点击"生成账单"
  → POST /api/bills/generate/{contract_id}  { bill_month: "2026-06" }
  → generate_bill()
    → 检查合同状态（必须是"已租"或"退租处理中"）
    → 检查是否已有同月账单（幂等保护）
    → 查询当月电表读数，计算电费
    → 计算 due_date = bill_month + contract.rent_due_day
    → INSERT INTO bills (..., due_date, status='未付')
```

### 2.4 约束保障

[`backend/models/orm.py`](backend/models/orm.py) 第 61 行：

```python
CheckConstraint("rent_due_day BETWEEN 1 AND 28", name="chk_contract_due_day"),
```

- `rent_due_day` 被约束在 1-28 之间，避免 29/30/31 在 2 月等短月份出问题
- `due_date` 存储为 TEXT 类型（格式 `YYYY-MM-DD`），与 SQLite 的 `date()` 函数返回格式一致，字符串比较即可正确判断先后

## 三、逾期判断：Dashboard API

### 3.1 代码位置

**文件**：[`backend/main.py`](backend/main.py)

**端点**：`GET /api/dashboard`（第 80-108 行）

### 3.2 执行逻辑

```python
# 第 87-92 行
overdue = db.scalar(
    select(func.count(Bill.id)).where(
        Bill.status.in_(("未付", "部分付")),
        Bill.due_date < func.date("now", "localtime"),
    )
) or 0
```

**SQL 等价**：
```sql
SELECT COUNT(*)
FROM bills
WHERE status IN ('未付', '部分付')
  AND due_date < date('now', 'localtime');
```

### 3.3 判断条件

逾期 = **同时满足**以下两个条件：
1. `status` 为 `"未付"` 或 `"部分付"` —— 账单尚未完全付清
2. `due_date < date('now', 'localtime')` —— 到期日已过（按本地时间）

### 3.4 首页展示链路

```
用户打开首页 / App.vue 切换"首页"Tab
  → Home.vue onMounted
  → api.get('/dashboard')
  → dashboard() 函数
  → 实时 SQL 查询 overdue_bill_count
  → 返回 {"overdue_bill_count": N, ...}
  → 前端显示 N 条逾期
```

[`frontend/src/views/Home.vue`](frontend/src/views/Home.vue) 对应展示：

```html
<van-grid-item text="逾期账单" :value="dashboard.overdue_bill_count" />
```

### 3.5 Dashboard 还统计了"今日应收"

```python
# main.py 第 96-102 行
today_day = date.today().day  # 今天几号
today_rent = db.scalar(
    select(func.coalesce(func.sum(Contract.monthly_rent), 0)).where(
        Contract.status.in_(("已租", "退租处理中")),
```

这是**直接查合同表**，不依赖账单是否已生成。用于首页展示"今日待收租金"。

## 六、催收机制

### 6.1 当前实现

**文件**：[`backend/services/bill_service.py`](backend/services/bill_service.py) 第 184-193 行

```python
def send_dunning(db: Session, bill_id: int, data) -> dict:
    b = db.get(Bill, bill_id)
    if not b:
        raise HTTPException(404, "账单不存在")
    if b.status == "已付":
        raise HTTPException(400, "账单已付，无需催收")
    db.add(DunningLog(bill_id=bill_id, sms_content=data.sms_content))
    db.commit()
    return get_bill(db, bill_id)
```

**完全手动**：用户在账单详情页点击"催收"按钮 → 填写短信内容 → POST 到 API → 仅写入 `dunning_logs` 表（不实际发送短信）。

### 6.2 缺失的自动化

| 应有的自动化 | 当前状态 |
|-------------|---------|
| 每天扫描逾期账单 | **不存在** — 无定时任务 |
| 自动标记"逾期"状态 | **不存在** — `Bill.status` 只有 未付/部分付/已付 |
| 自动发送催收短信 | **不存在** — `send_dunning` 不调短信接口 |
| 逾期账单列表/筛选 | **不存在** — 前端无"逾期"Tab |

## 十、总结

| 维度 | 现状 | 评估 |
|------|------|------|
| 到期日计算 | `due_date = bill_month + contract.rent_due_day` | ✅ 正确 |
| 逾期判断逻辑 | `due_date < date('now','localtime') AND status IN ('未付','部分付')` | ✅ 正确 |
| 每日自动检测 | 无定时任务 | ❌ 缺失 |
| 逾期状态持久化 | 数据库无"逾期"状态 | ⚠️ 设计选择（实时计算也有好处） |
| 自动催收 | 无 | ❌ 缺失 |
| Dashboard 逾期展示 | 每次打开首页实时查询 | ✅ 工作正常 |
| 租客信用逾期统计 | 同样通过 `due_date < now` 查询 | ✅ 工作正常 |

**最终结论**：系统**能正确判断逾期**，到期日根据合同截止日计算，判断逻辑准确。但**不会每天自动检测**——逾期统计完全依赖用户打开首页时 Dashboard API 的被动查询。如果需要每天自动扫描、自动标记、自动催收，需要额外开发定时任务模块。
